**场景：AI 助手的「对话记忆」怎么存？**

假设我们在搞一个对标豆包的 AI 助手，核心功能之一是**多轮对话记忆**——用户今天聊完，明天回来还能接着聊。现在有三种方案摆在我们面前：

---

### 方案 A：全量存，整条对话当一条记录

| 维度 | 说明 |
|------|------|
| 做法 | 每次对话完整存为一条 JSON，按 session_id 索引 |
| 存储 | Redis / MongoDB |
| 优点 | 实现简单，检索快，适合短对话 |
| 缺点 | 对话越长，存储越大；要做上下文截断时得自己拼逻辑 |
| 维护成本 | 低 |
| 扩展性 | 一般，长对话会膨胀 |

### 方案 B：按「消息」粒度存，按时间戳排序

| 维度 | 说明 |
|------|------|
| 做法 | 每条消息独立存储（role + content + timestamp），按 session_id 查询时拼回 |
| 存储 | PostgreSQL + 时间序列索引 / Cassandra |
| 优点 | 灵活，可以做细粒度的编辑/删除/搜索 |
| 缺点 | 查询时需要聚合，拼回完整上下文的开销存在 |
| 维护成本 | 中 |
| 扩展性 | 好，适合大规模并发 |

### 方案 C：向量化存，只记「语义」

| 维度 | 说明 |
|------|------|
| 做法 | 把每轮关键信息向量化存入向量数据库，对话时不靠全文，靠语义检索 |
| 存储 | Chroma / Milvus / Qdrant |
| 优点 | 节省 token，能跨越多个 session 做联想，天然适合 RAG |
| 缺点 | 向量召回有精度损失，冷启动还得搭配方案 B 兜底 |
| 维护成本 | 中高（需要搭配 embedding 服务） |
| 扩展性 | 最强，但架构复杂度最高 |

### 方案 D：分层记忆架构（Hierarchical Memory）

**核心思想：** 不让任何一层记忆做所有事，拆成四层，各司其职。

| 层级 | 名称 | 做什么 | 存哪 | 多久 |
|------|------|--------|------|------|
| **L1** | 工作记忆 | 当前对话的上下文滑动窗口（最近 N 轮） | 内存 | 会话期间 |
| **L2** | 短期记忆 | 最近几轮完整消息，支持回溯 | Redis / 本地缓存 | 几个小时～几天 |
| **L3** | 长期记忆 | 经 LLM **自动摘要压缩**后的历史对话要点 | PostgreSQL / SQLite | 永久（压缩态） |
| **L4** | 核心记忆 | 用户关键事实、偏好、决策记录，结构化存储 | 关系型 DB | 永久（高优先级） |

**工作流程：**

```
用户说了一句话
  ↓
L1 滑动窗口命中 → 直接回复
  ↓ (滑动窗口溢出)
L2 短期记忆接手，补充上下文
  ↓ (对话结束 / 超过 N 轮)
LLM 自动把本轮对话压缩成 1-3 句摘要，存入 L3
  ↓ (LLM 识别到关键信息)
关键事实（名字、偏好、承诺等）结构化存入 L4
```

**优点：**
- ✅ **性能好** — 90% 的查询 L1/L2 就搞定了，不用查大存储
- ✅ **Token 省** — 历史对话不是全量拼回，而是压缩后的摘要
- ✅ **精度高** — 核心事实是结构化存的，不会"记错"
- ✅ **可解释** — 你知道每一层存的是什么

**缺点：**
- ⚠️ 架构复杂度比 A/B/C 都高
- ⚠️ 需要 LLM 参与摘要生成（调度开销）
- ⚠️ L4 的结构化 schema 需要提前设计好

---
